# Emphant Studio 产品调整 TODO

## 背景

当前产品方向是本地优先的桌面 AI 工作台，功能覆盖对话、智能体、技能、知识库、任务、邮件和工具调用。现阶段主要问题不是功能不足，而是核心概念、动作审批、数据归属和运行状态不够清晰，导致用户在授权 AI 执行真实工作时缺少确定感。

本 TODO 用于后续产品和研发拆分，优先解决“可信、可控、可追踪”的核心体验。

## 术语约定

| 概念 | 定义 | 主要出现位置 |
| --- | --- | --- |
| 会话 | 用户与 AI 围绕一个上下文展开的对话容器，包含消息、已选智能体和当前会话工作目录。 | 首页左侧列表、对话工作台 |
| 任务 | 可执行、可追踪、有状态的工作项，支持手动执行、定时执行和结果回写。 | 任务页面 |
| 执行记录 | AI 或后台任务针对某个动作的运行过程、结果、错误和审批痕迹。 | 后续动作审批、运行详情、后台任务模块 |
| 智能体 | 具备角色、模型、知识范围、技能和工具权限的执行者。 | 智能体页面、会话调度 |
| 技能 | 可被智能体调用的提示词或代码能力模块，用于增强特定任务表现。 | 技能页面、智能体能力配置 |
| 工具 | 可访问外部系统或本机能力的操作接口，例如搜索、文件、命令、数据库和邮件。 | 设置页、智能体工具挂载、动作审批 |
| 知识库 | 可被 AI 检索和引用的资料集合，包含原始文件、切片、图谱和引用来源。 | 知识页面、会话引用 |

## P0 必须优先处理

- [x] 统一核心术语
  - [x] 明确定义“会话”“任务”“执行记录”“智能体”“技能”“工具”“知识库”。
  - [x] 将首页左侧的“任务”改为“会话”。
  - [x] 删除会话、删除任务、删除待办等提示文案分别区分。
  - [x] 检查全站文案，避免“任务 / 会话 / 待办 / 执行”混用。

- [ ] 重构首页主路径
  - [x] 首页聚焦“对话 + 当前上下文 + 可用能力 + 待确认动作”。
  - [ ] 将邮件、任务生成、语音、知识引用等能力从发送逻辑中拆成可插拔模块或侧边面板。
  - [x] 增加当前会话的上下文摘要，包括已选智能体、知识库、工具权限和工作目录。
  - [x] 普通聊天场景不展示无关的邮件、任务和工具配置干扰。

- [x] 建立统一 AI 动作审批机制
  - [x] 所有高风险动作先生成结构化动作，再进入审批队列。
  - [x] 发邮件、写文件、执行命令、数据库写操作、删除数据必须确认。
  - [x] 审批卡片展示动作类型、执行智能体、目标资源、具体参数、风险说明。
  - [x] 支持“允许”“拒绝”“修改后执行”。
  - [x] 所有审批结果写入执行历史。

- [x] 替换前端正则意图判断
  - [x] 移除前端对邮件发送、邮件总结等动作的正则直判逻辑。
  - [x] 引入统一 Intent Planner，输出结构化意图、置信度、参数和是否需要确认。
  - [x] 前端只负责展示结构化意图和确认动作，不直接决定高风险行为。

- [x] 修复登录态和本地安全机制
  - [x] localStorage 不保存明文、base64 或其他可逆密码。
  - [x] 登录态、保险箱密码、API Key、邮箱凭据统一进入系统安全存储或后端安全模块。
  - [x] 增加手动锁定保险箱能力。
  - [x] 高风险操作支持二次确认。

- [x] 明确数据归属和保存位置
  - [x] 区分系统预设、用户数据、本地安全数据、工作目录数据和临时数据。
  - [x] 在设置中增加“数据管理”区域。
  - [x] 展示每类数据的保存位置、可导出状态和清除入口。
  - [x] 系统预设不能和用户资产混在一起，升级合并前需要明确来源。

## 验收总目标

- [x] 用户能清楚知道当前是在对话、创建任务、执行任务还是配置能力。
- [x] 用户能清楚知道 AI 准备执行什么动作、访问什么资源、是否需要确认。
- [x] 用户能追踪每次 AI 回复使用了哪些模型、知识和工具。
